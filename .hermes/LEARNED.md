### 5. Risk check — three real production smells in the last 5 commits

### 3. Code check — duplicate limiter pattern is now 3 copies, RegoErrorEnvelope still inline x6 (NOTE only)
- **The hourly limiter is now triplicated and diverging.** `src/lib/qld-rego/official.ts` uses `hourlyHits: number[]` (array). `src/app/api/rego/capture/route.ts` (iter 24) used `captureHourlyHits: number[]` (same shape). Flagged third review running. **Do NOT refactor in the loop — flag for awake-Jordan.**
- **`captureHourlyHitsByEmail` is unbounded.** Iter 28's Map has no eviction of empty arrays after the shift loop empties them, and no cap on distinct keys. A long-lived instance processing many distinct emails grows the Map indefinitely; the per-email arrays drain over the window but the Map entries don't. Low real risk on Vercel (instances are short-lived) but it's there. Worth flagging because the abuse vector for the limiter is now "spray distinct emails until the map gets fat" — small attack, but the limiter exists *because* abuse is being modelled.
- **Iter 27 added a sixth hand-rolled envelope** (`routeUnhandledResponse`). Status is `"error"`, error field is `"route_unhandled"`. Flagged third review running.
- **`STATUS_TO_HTTP` is genuinely good.** The `satisfies Record<QldRegoCheckFailure["status"], number>` clause is the right pattern — exhaustiveness checked at typecheck time, not at runtime. ### 5. Risk check — three smells, one new from this batch
- **(NEW, low–medium) Iter 26's reorder traded one partial-success state for another.** Old order: notification fails after buyer email sent → buyer has email, Jordan blind. New order: buyer email fails after notification sent → Jordan has lead, buyer never gets follow-up. Better outcome (Jordan can manually re-send to the buyer; Jordan was the missing party before), but **there is no log line tagged `partial_success` and no retry of the buyer send**. If Resend has a transient buyer-side bounce while notification succeeds, the public response is `provider_error` but Jordan's notification email lands. From Jordan's inbox it looks like a normal lead; from the user's UI it looks like a failure; both reload and retry, doubling Jordan's leads and still failing the buyer send. **Not blocking — but the partial-success state is still real and now silently shifted to the harder-to-detect direction**. Either tag the partial state in the log (`console.warn("rego capture buyer-send failed after notification succeeded", { regoNoLeak, emailNoLeak })`) or retry-once before returning provider_error.
- **(NEW, low) `captureHourlyHitsByEmail` Map grows monotonically.** Per-instance + unbounded distinct keys = memory leak. Empty buckets aren't pruned. Real cap = instance lifetime × email-rate. Vercel Fluid Compute keeps instances warmer than classic serverless (per session-context: instance reuse is the model), so this matters slightly more on Vercel than on plain AWS Lambda. Trivial fix (`if (hits.length === 0) captureHourlyHitsByEmail.delete(key)` after shifts) — flag only.
- **(carryover, still standing, do NOT pick in the loop) Per-instance limit on Fluid Compute is still functionally a per-instance allowance × N instances, not a true per-caller cap.** Disclosed via `x-rego-rate-limit-scope: instance`. The real fix is Vercel Runtime Cache or a Marketplace KV (Upstash Redis) — awake-Jordan task, not loop work. The loop should *stop polishing* the per-instance limiter (per-email keys, env-var tuning, etc.) and accept the per-instance scope as fixed until the storage choice is made.
- **(carryover, still standing) `escapeHtml(unknown)` signature is still load-bearing.** Iter 23 narrowed `reason` to `string | undefined` at the route level, but `escapeHtml(value: unknown)` remains wide elsewhere. Same flag as last review.
